# Observabilidade

- Rule ID: R-O11Y-001
- Severidade: hard para dados sensíveis e visibilidade de erros, guideline para granularidade.
- Escopo: Handlers, use cases, repositórios e adapters de infraestrutura.

## Objetivo
Garantir logging, tracing e métricas consistentes.

## Requisitos

### Provedor e Injeção (hard)
- Usar a abstração de observabilidade do projeto.
- Injetar provedor de observabilidade via construtor.
- Usar provedor fake em testes.

### Tracing
- **(hard)** Sempre encerrar spans e registrar erros.
- **(guideline)** Métodos públicos em handlers/use cases/repositórios devem abrir spans.
- **(guideline)** Nomes de span devem seguir `{entity}_{layer}.{operation}`.

### Logging
- **(hard)** Usar apenas logging estruturado.
- **(hard)** Não logar PII ou segredos (ver R-SEC-001).
- **(guideline)** Campos mínimos: `operation`, `layer`, `entity` (nome do bounded context, ex.: `budget`, `transaction`).
- **(guideline)** Logs de handler devem incluir `correlation_id` e `user_id` quando disponíveis.

### Métricas (guideline)
- Usar instrumentos OpenTelemetry através da abstração compartilhada.
- Usar nomenclatura consistente: `{service}.{module}.{metric_name}`.

## Proibido
- Logar PII/segredos.
- Logs não estruturados para fluxo de aplicação.
- Ausência de registro de erro em traces onde erros ocorrem.
- `fmt.Println`/`log.Println` para logging de aplicação.
