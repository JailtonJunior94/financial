# Observability

## Purpose
Enforce consistent logging, tracing, and metrics patterns across the codebase.

## Scope
All layers: handlers, use cases, repositories, and infrastructure adapters.

## Requirements

### Observability Provider
- Must use `devkit-go/pkg/observability` as the single observability abstraction.
- Must inject `observability.Observability` via constructor in all layers.
- Must use `fake.NewProvider()` in tests.

### Tracing
- Every public method in handlers, use cases, and repositories must start a span.
- Span naming convention: `{entity}_{layer}.{operation}` (e.g., `category_handler.create`, `category_repository.save`, `create_category_usecase.execute`).
- Must always defer `span.End()`.
- Must record errors on spans via `span.RecordError(err)`.
- Must add events for meaningful state changes via `span.AddEvent(...)`.

### Logging

#### Levels
| Level | Usage |
|-------|-------|
| Debug | Repository query start/complete, internal state transitions |
| Info  | Handler request received/completed |
| Warn  | Expected failures (not found, validation) |
| Error | Unexpected failures (database errors, infrastructure errors) |

#### Structure
- Must always use structured logging with `observability.String`, `observability.Error`.
- Every log entry must include: `operation`, `layer`, `entity`.
- Handler logs must also include: `correlation_id`, `user_id`.
- Repository logs must also include: `user_id` and relevant entity IDs.

#### Output
- Logs must go to stdout/stderr only.

### Metrics
- Must use OpenTelemetry instruments via `observability.Metrics()`.
- Module-specific metrics must be grouped in a dedicated struct (e.g., `FinancialMetrics`, `CardMetrics`).
- Metric naming convention: `financial.{module}.{metric_name}`.
- Repository operations must record query duration via `RecordRepositoryQuery`.
- Repository failures must record failure metrics via `RecordRepositoryFailure`.

### Sensitive Data
- Must never log PII: names, emails, addresses, credit card numbers, passwords.
- Must log only identifiers: `user_id`, `category_id`, `order_id`.
- Must never add PII to span attributes or events.

### Log Messages
- Must be concise and machine-parseable.
- Must use snake_case for log event names: `request_received`, `query_failed`, `query_started`, `query_completed`, `request_completed`.

## Examples

### Tracing
```go
ctx, span := u.o11y.Tracer().Start(ctx, "create_category_usecase.execute")
defer span.End()
```

### Structured Logging
```go
u.o11y.Logger().Info(ctx, "request_received",
    observability.String("operation", "CreateCategory"),
    observability.String("layer", "handler"),
    observability.String("entity", "category"),
    observability.String("correlation_id", correlationID),
    observability.String("user_id", user.ID),
)
```

### Repository Metrics
```go
r.fm.RecordRepositoryQuery(ctx, "list", "category", time.Since(start))
r.fm.RecordRepositoryFailure(ctx, "list", "category", "infra", time.Since(start))
```

### Sensitive Data
```go
// Forbidden
observability.String("email", user.Email)
observability.String("name", user.Name)

// Required
observability.String("user_id", user.ID)
```

## Forbidden
- Logging PII (names, emails, addresses, credit cards, passwords, SSN).
- Silent error swallowing.
- Unstructured string interpolation in logs.
- Missing spans in public methods of handlers, use cases, or repositories.
- Logging to files.
- Using `fmt.Println` or `log.Println` for application logging.
